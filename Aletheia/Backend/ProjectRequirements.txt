# Aletheia — Low-Level Requirements (MVP First)

## Product Overview

### App Name

**Aletheia**

### Core Purpose

A web-based library tracking and borrowing platform that allows students, teachers, researchers, and book enthusiasts to:

* Search books
* Check availability
* Borrow/reserve books
* Track return schedules
* Securely authenticate using school email + MFA

---

# 1. MVP Scope (Highest Priority Features)

The MVP should focus only on the features necessary to make the system functional, secure, and demonstrable.

## MVP Goals

A user should be able to:

1. Register using NU Laguna email
2. Verify email
3. Login securely
4. Search books
5. View book availability
6. Borrow a book
7. View borrowed books
8. See return deadlines
9. Return books

# 5. Functional Requirements (Prioritized)

# PRIORITY 1 — Authentication Module

This is the foundation of the entire system.

---

## 5.1 User Registration

### Description

Users register using their NU Laguna email.

### Requirements

* Email must end with:

```text
@nu-laguna.edu.ph
```

* Required fields:

  * First Name
  * Last Name
  * School ID
  * Email
  * Password

### Validation Rules

* Email must be unique
* School ID must be unique
* Password minimum 8 characters
* Password must contain:

  * uppercase
  * lowercase
  * number

---

## 5.2 Email Verification

### Flow

1. User registers
2. System sends OTP/code to Outlook email
3. User enters code
4. Account becomes active

### Requirements

* OTP expires in 5 minutes
* User can resend OTP
* Account cannot login unless verified

---

## 5.3 Multi-Factor Authentication (MFA)

### Flow

1. User logs in
2. System validates password
3. User enters authenticator OTP

### MVP Simplification

Use:

* TOTP authenticator apps

  * Google Authenticator
  * Microsoft Authenticator

### Requirements

* QR generation during MFA setup
* OTP expires every 30 seconds

---

## 5.4 Login

### Requirements

* JWT-based authentication
* Secure password hashing using BCrypt
* Sessionless backend

### Failed Login Rules

* 5 failed attempts lock account for 15 minutes

### Available

If:

```text
availableCopies > 0
```

### Unavailable

If:

```text

```

# PRIORITY 3 — Borrowing System

---

## 7.1 Borrow Book

### Flow

1. User selects book
2. System checks availability
3. Borrow record created
4. availableCopies decreases

### Requirements

* One user may borrow maximum 5 books
* Borrow period:

```text
14 days
```

* Cannot borrow unavailable books

---

## 7.2 Borrow Record Entity

| Field      | Type     |
| ---------- | -------- |
| id         | Long     |
| userId     | Long     |
| bookId     | Long     |
| borrowDate | DateTime |
| dueDate    | DateTime |
| returnDate | DateTime |
| status     | Enum     |

### Status Enum

```text
BORROWED
RETURNED
OVERDUE
RESERVED
```

---

## 7.3 Return Book

### Flow

1. User/admin marks book returned
2. Return date stored
3. availableCopies increases

---

## 7.4 Overdue Detection

### Rule

If:

```text
currentDate > dueDate
```

Status becomes:

```text
OVERDUE
```
